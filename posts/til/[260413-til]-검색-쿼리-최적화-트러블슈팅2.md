<h1>텍스트 검색 쿼리 1초 &rarr; 0.6ms 최적화 삽질기 (2탄)</h1>
<h2 data-ke-size="size26">TL;DR</h2>
<p data-ke-size="size16"><a href="https://ifelseif.tistory.com/332">1탄</a>에서 10초 &rarr; 1초까지 줄였던 검색 쿼리가, 사실 pg_trgm GIN 인덱스를 <b>전혀 타지 않고 있었다.</b> EXPLAIN ANALYZE로 실행 계획을 직접 확인한 결과 매번 Seq Scan(풀스캔)이 돌고 있었고, 검색 방식을 ILIKE &rarr; 단어 배열(<code>text[]</code>) + GIN <code>array_ops</code>로 교체하여 최종 0.6ms까지 단축했다.</p>
<h2 data-ke-size="size26">이전 글 요약</h2>
<ul style="list-style-type: disc;" data-ke-list-type="disc">
<li>MV + <code>search_text</code> 컬럼(caption + hashtags 결합) + <code>pg_trgm</code> GIN 인덱스로 1초 달성</li>
<li>이 시점에서 "GIN 인덱스 덕에 빨라졌다"고 결론 내림</li>
<li><b>이 결론이 완전히 틀렸다.</b></li>
</ul>
<h2 data-ke-size="size26">삽질기: 1초 &rarr; 0.6ms까지의 여정</h2>
<h3 data-ke-size="size23">7단계: 아직도 5초?</h3>
<p data-ke-size="size16">1탄에서 1초를 달성했다고 썼지만, 사실 프론트에서 실제로 치는 쿼리 패턴(검색어 + 정렬 + 페이지네이션 조합)에서는 여전히 <b>5-6초</b>가 걸리고 있었다.</p>
<h3 data-ke-size="size23">8단계: EXPLAIN ANALYZE를 처음 찍어보다</h3>
<p data-ke-size="size16">이전까지는 "인덱스를 걸었으니 탈 것이다"라고 믿고 있었다. 이번에 처음으로 <code>EXPLAIN (ANALYZE, BUFFERS)</code>를 찍어봤다.</p>
<pre class="pgsql"><code>EXPLAIN (ANALYZE, BUFFERS)
SELECT *
FROM public.reel_growth_rate_economy g
WHERE g.search_text ILIKE '%경제%'
ORDER BY g.likes DESC NULLS LAST
LIMIT 20;</code></pre>
<p data-ke-size="size16">결과:</p>
<pre class="pgsql"><code>Index Scan using idx_reel_growth_rate_economy_likes
  Filter: (search_text ~~* '%경제%'::text)
  Rows Removed by Filter: 4408
  Buffers: shared hit=7638</code></pre>
<p data-ke-size="size16">&nbsp;</p>
<p data-ke-size="size16"><b>GIN trigram 인덱스가 아니라 likes B-tree 인덱스를 타고 있었다.</b> 플래너가 <code>ORDER BY likes DESC LIMIT 20</code>을 보고 "likes 순으로 스캔하면서 ILIKE 조건 맞는 20개만 찾으면 되겠다"고 판단한 것이다. 문제는 "경제"가 전체 4.2만 행 중 1,538행(3.6%)에만 매칭되기 때문에, 20개를 채우려면 <b>거의 전체를 스캔</b>해야 한다는 것.</p>
<p data-ke-size="size16">ILIKE 조건은 인덱스가 아닌 <b>필터(Filter)</b> 로 적용되고 있었다.</p>
<h3 data-ke-size="size23">9단계: 그러면 GIN 인덱스를 강제로 태우면?</h3>
<p data-ke-size="size16"><code>SET enable_seqscan = off;</code>로 강제로 GIN 인덱스를 태워봤다.</p>
<pre class="lasso"><code>Bitmap Index Scan on idx_reel_growth_rate_economy_search_text
  Index Cond: (search_text ~~* '%경제%'::text)
  &rarr; 47,368 rows
Bitmap Heap Scan
  Recheck Cond: ...
  Rows Removed by Index Recheck: 40,720</code></pre>
<p data-ke-size="size16">&nbsp;</p>
<p data-ke-size="size16">GIN 인덱스가 <b>47,368행</b>을 반환했다. MV 전체(42,258행)보다 많다. 인덱스 레벨에서 거의 모든 행이 후보로 잡히고, Heap에서 실제 ILIKE를 다시 검증해서 대부분 버리는 구조. <b>Seq Scan보다 더 느렸다(9초 &rarr; 21초).</b></p>
<h3 data-ke-size="size23">10단계: pg_trgm의 근본적 한계를 이해하다</h3>
<p data-ke-size="size16">원인은 <b>한글 2글자 검색어와 trigram의 궁합</b>이었다.</p>
<p data-ke-size="size16"><code>pg_trgm</code>은 문자열을 3글자(trigram) 단위로 쪼개서 인덱싱한다. </p>
<p data-ke-size="size16"><br />"경제"는 2글자라 생성되는 trigram이 극히 적고, 인덱스 레벨에서의 필터링 효과가 거의 없다. <br />결과적으로 인덱스를 탄다 해도 후보군이 전체와 비슷하게 나와서 의미가 없었던 것이다.<br /><br /></p>
<p data-ke-size="size16"><b>1탄의 결론이었던 "pg_trgm GIN 인덱스 덕에 1초를 달성했다"는 완전히 틀린 결론이었다.</b> <br />실제로는 MV 자체의 효과(매번 JOIN &rarr; 미리 계산된 테이블 조회)와 다른 B-tree 인덱스들 덕이었지, <br />trigram 인덱스는 처음부터 아무 역할도 하지 않고 있었다.</p>
<h3 data-ke-size="size23">11단계: ILIKE를 버리고 단어 배열로 전환</h3>
<p data-ke-size="size16">ILIKE 부분 매칭을 포기하고, <b>공백 기준으로 단어를 쪼개서 배열(<code>text[]</code>)로 저장</b> + <b>GIN <code>array_ops</code> 인덱스</b>로 exact match 검색으로 전환했다.</p>
<pre class="pgsql"><code>-- MV에 search_words 컬럼 추가
array(
  select distinct w
  from unnest(string_to_array(lower(caption || ' ' || hashtags), ' ')) as w
  where w &lt;&gt; ''
) as search_words
<p>-- GIN 인덱스
CREATE INDEX idx_search_words
ON reel_growth_rate_economy USING gin (search_words array_ops);</p>
<p>-- 쿼리: &quot;이 배열에 '경제'라는 원소가 있는가&quot;
WHERE search_words &amp;&amp; ARRAY['경제']::text[]</code></pre></p>
<p data-ke-size="size16">결과: <b>0.56ms.</b> 기존 대비 약 16,000배 빨라졌다.</p>
<h3 data-ke-size="size23">12단계: 결과가 절반으로 줄었다?</h3>
<p data-ke-size="size16">그런데 검색 결과가 1,200건 &rarr; 675건으로 줄었다.<br /><br /></p>
<p data-ke-size="size16">원인은 명확했다. 기존 ILIKE <code>'%경제%'</code>는 부분 매칭이라 "경제뉴스를", "한국경제", "경제적" 등이 모두 잡혔지만, <br />배열 exact match에서는 정확히 "경제"라는 단어만 매칭된다. 한글은 조사가 붙고 합성어가 많아서 공백 기준 토큰화의 한계가 있다.<br /><br /></p>
<p data-ke-size="size16">서브스트링(N-gram)을 배열에 넣는 방법도 검토했지만, 평균 57개인 배열이 수백 개로 불어나기 때문에 비현실적이었다. 결국 exact match로 가되 프론트에서 검색 가이드를 제공하는 방향으로 결정했다.</p>
<h2 data-ke-size="size26">교훈</h2>
<h3 data-ke-size="size23">"인덱스를 걸었으니 타겠지"는 위험한 가정이다</h3>
<p data-ke-size="size16">인덱스를 만드는 것과 인덱스가 실제로 사용되는 것은 완전히 다른 문제다. <b>EXPLAIN ANALYZE를 찍어보기 전까지는 인덱스가 타는지 알 수 없다.</b> 이번에 pg_trgm GIN 인덱스를 만들어놓고 한 번도 확인하지 않은 채 "덕분에 빨라졌다"고 결론 내렸던 게 대표적인 사례다.</p>
<h3 data-ke-size="size23">플래너는 나보다 똑똑하지만, 내 의도를 모른다</h3>
<p data-ke-size="size16">PostgreSQL 쿼리 플래너는 통계를 기반으로 가장 비용이 낮은 실행 계획을 선택한다. <code>ORDER BY likes DESC LIMIT 20</code> + <code>ILIKE</code> 조건이 있으면, 플래너는 "likes 인덱스로 정렬 순서를 공짜로 얻고, 필터로 걸러내자"고 판단한다. 매칭 비율이 높으면 이 전략이 효율적이지만, 매칭 비율이 낮으면(이번 경우 3.6%) 사실상 풀스캔이 된다. 플래너의 선택이 항상 최선은 아니다.</p>
<h3 data-ke-size="size23">pg_trgm은 만능이 아니다 &mdash; 특히 한글 짧은 검색어에서</h3>
<p data-ke-size="size16">pg_trgm은 3글자 단위로 인덱싱한다. 검색어가 2글자("경제")면 생성되는 trigram이 적어서, 인덱스 레벨에서 후보군을 좁히지 못한다. "ILIKE가 느리면 pg_trgm을 쓰면 된다"는 공식이 항상 성립하는 건 아니다. 데이터의 언어, 검색어 길이, 매칭 비율에 따라 전혀 다른 전략이 필요할 수 있다.</p>
<h3 data-ke-size="size23">인덱스 전략은 데이터 특성에서 출발해야 한다</h3>
<p data-ke-size="size16">이번에 효과적이었던 접근:</p>
<ol style="list-style-type: decimal;" data-ke-list-type="decimal">
<li><b>EXPLAIN ANALYZE로 현재 실행 계획 확인</b> &mdash; 추측이 아닌 사실 기반</li>
<li><b>인덱스를 강제로 태워서 비교</b> &mdash; <code>SET enable_seqscan = off</code></li>
<li><b>인덱스가 비효율적인 이유를 이해</b> &mdash; trigram + 한글 2글자의 구조적 한계</li>
<li><b>검색 요구사항 재정의</b> &mdash; "부분 매칭이 꼭 필요한가?" &rarr; exact match로 충분</li>
<li><b>데이터 구조 자체를 변경</b> &mdash; ILIKE 문자열 검색 &rarr; 단어 배열 + GIN array_ops</li>
</ol>
<p data-ke-size="size16">"어떤 인덱스를 걸까"보다 먼저, "이 데이터에서 이 검색 패턴이면 어떤 자료구조가 맞는가"를 고민해야 한다.</p>
<h3 data-ke-size="size23">EXPLAIN ANALYZE는 DB 작업의 <code>console.log</code>다</h3>
<p data-ke-size="size16">프론트에서 뭔가 이상하면 콘솔부터 열 듯이, DB 쿼리가 느리면 EXPLAIN ANALYZE부터 찍어야 한다. 이번에 처음 찍어봤는데, 진작 했으면 1탄에서 "pg_trgm 덕에 빨라졌다"는 오진을 하지 않았을 것이다.</p>
